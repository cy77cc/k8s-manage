## Context

k8s-manage 当前功能覆盖较完整，但品牌识别和界面一致性不足：产品名称缺乏记忆点、Logo 体系缺失、页面视觉与交互模式不统一，导致用户在跨模块操作时学习成本高。该变更是一次跨前端基础设施、页面信息架构和体验规范的横切改造，涉及 `web/src/layout`、`web/src/components`、`web/src/pages`、`web/src/styles` 以及可能的配置读取入口（如品牌配置）。

约束包括：
- 不破坏现有 `/api/v1` 业务契约；
- 保留并强化基于角色的导航/操作可见性约束；
- 通过渐进式上线降低大规模 UI 重构风险。

## Goals / Non-Goals

**Goals:**
- 交付新的平台品牌名称与 Logo 体系，并定义在系统中的统一应用规则。
- 建立可复用的 UI 设计基线（颜色、字体、间距、圆角、阴影、状态语义等 tokens）。
- 统一全局布局与导航结构，优化核心任务路径（查看、筛选、编辑、确认）和反馈机制。
- 使权限治理相关页面在新视觉语言下保持高可读性和高可发现性。

**Non-Goals:**
- 不重写后端核心业务逻辑，不引入大规模新领域模型。
- 不在本次变更中完成所有页面像素级彻底重绘，优先覆盖全局骨架与核心高频页面。
- 不替换现有 RBAC/Casbin 机制，仅改进其在 UI 的呈现与交互可解释性。

## Decisions

### Decision 1: 品牌体系采用“名称 + 主标 + 简化标 + 单色版”四件套
- Rationale: 同时满足导航栏、登录页、favicon、文档和深浅背景场景，减少后续临时出图。
- Alternatives considered:
  - 仅提供单一 Logo：交付快，但跨场景适配差。
  - 先不改名只改图标：无法解决品牌识别问题。

### Decision 2: 用 Design Tokens 驱动 UI 重构
- Rationale: 把视觉决策沉淀到 `web/src/styles` 与主题配置，避免页面内散落硬编码样式。
- Alternatives considered:
  - 页面逐个直接改 CSS：短期可见，长期不可维护。
  - 完全替换 UI 库：成本和风险过高。

### Decision 3: 导航 IA 调整为“任务导向分组 + 角色感知展示”
- Rationale: 让用户按目标完成任务，而不是按系统实现细节找入口；继续遵循权限可见性边界。
- Alternatives considered:
  - 保持原导航仅换皮：不能解决可发现性问题。
  - 彻底个性化导航：复杂度高，首轮不做。

### Decision 4: 交互统一采用“显式入口 + 分层反馈 + 安全确认”
- Rationale: 避免隐藏操作路径，关键变更动作统一提供结果反馈与确认步骤，降低误操作。
- Alternatives considered:
  - 延续各页面既有交互：一致性不足。
  - 全量改为向导式流程：会降低高频操作效率。

### Decision 5: 采用分阶段发布
- Rationale: 先落地主题与全局框架，再覆盖关键模块，降低一次性发布风险。
- Alternatives considered:
  - 大爆炸式切换：回滚成本高。

## Risks / Trade-offs

- [Risk] 视觉重构导致用户短期不适应 → Mitigation: 发布更新说明、保留关键入口命名一致、提供过渡期提示。
- [Risk] 页面重构期间出现样式回归与交互不一致 → Mitigation: 建立 tokens/组件验收清单并做关键页面回归测试。
- [Risk] 导航调整可能影响角色可见性规则 → Mitigation: 将导航可见性校验纳入角色维度测试矩阵。
- [Risk] 品牌素材在不同尺寸下识别度不足 → Mitigation: 交付主标/简化标/单色版并进行多尺寸可读性评审。

## Migration Plan

1. Phase 1（基础）：引入品牌资产、主题 tokens、全局布局容器与导航骨架。
2. Phase 2（核心）：重构高频页面与权限治理页面，确保核心任务路径完整。
3. Phase 3（收敛）：统一细节交互、补齐空状态/错误态、清理遗留样式。
4. Rollback: 保留旧主题与旧导航配置开关，出现严重问题时可回切至旧 UI 壳层。

## Open Questions

- Logo 资产是否需要同步输出品牌手册（最小尺寸、安全区、禁用示例）？
- 是否要求首期覆盖移动端适配，还是先完成桌面端后再扩展？

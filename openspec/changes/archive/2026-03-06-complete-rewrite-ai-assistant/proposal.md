# Proposal: AI 助手完全重构

## Summary

完全重构 AI 助手模块，采用 Ant Design X 重写前端、eino 混合模式重构后端 Agent，实现高质量的运维对话体验。

## Motivation

### 当前问题

| 问题 | 表现 | 影响 |
|------|------|------|
| AI 响应质量差 | 经常输出 JSON 计划而非执行，意图理解错误 | 用户无法完成预期操作 |
| 代码架构混乱 | 前端单文件 71KB，后端 47 个路由端点 | 维护困难，Bug 难以定位 |
| 性能问题 | 响应慢、资源消耗大 | 用户体验差 |
| 前端体验差 | 界面交互不流畅，信息展示不清晰 | 用户满意度低 |

### 根本原因

1. **架构演进失控**: 从简单对话 → 命令桥 → 工具调用 → Agent 模式，每次都在原有基础上叠加，未做根本性重构
2. **职责边界模糊**: 前端承担过多业务逻辑，后端路由粒度过细
3. **工具设计冗余**: 30+ 工具定义，很多功能重叠，Agent 难以选择正确工具
4. **缺少意图分层**: 所有请求都走完整的 Agent 流程，即使简单问答也需要规划执行

### 解决方案

```
当前架构                              目标架构
─────────────────────────────────    ─────────────────────────────────
前端: ChatInterface.tsx (71KB)       前端: Ant Design X 原子组件
后端: 47 个路由端点                   后端: 6 个核心端点
Agent: 单一 ReAct Agent              Agent: 混合模式 (简单问答 + Agent 执行)
工具: 30+ 独立工具                    工具: 10 个聚合工具
审批: 中断+数据库轮询                 审批: 交互式询问+确认
存储: 内存 map                        存储: DB 持久化 + Redis 缓存
```

## Scope

### In Scope

1. **前端重写**
   - 使用 `@ant-design/x` 组件库重写 ChatInterface
   - 实现交互式确认 UI (类似 AskUserQuestion)
   - 流式 Markdown 渲染 + 代码高亮 + Mermaid 图表
   - 会话管理、历史记录

2. **后端简化**
   - 合并 47 个路由到 6 个核心端点
   - 统一 SSE 事件格式
   - 移除冗余 handler 逻辑

3. **Agent 重构**
   - 实现混合模式：意图分类 → 简单问答/Agent 执行
   - 交互式确认机制（高风险操作主动询问）
   - 保留 MCP 支持

4. **工具简化**
   - 合并相关工具（如 k8s_list_pods + k8s_list_services → k8s_query）
   - 统一工具注册和元数据管理
   - 保留核心能力：K8s、主机、部署、监控

5. **存储重构**
   - 会话消息持久化到数据库
   - Agent 检查点存储到 Redis
   - 移除内存 map 状态存储

### Out of Scope

- AI 模型切换（保持 eino 框架）
- 多租户隔离（后续变更）
- 权限系统变更（保持现有 RBAC）
- RAG 知识库重构（后续变更）

## Key Decisions

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 前端框架 | Ant Design X | 专为 AI 界面设计，提供 Sender/Bubble/Conversation 等原子组件 |
| Agent 模式 | 混合模式 | 简单问答直接响应，复杂任务走 Agent，避免不必要的规划开销 |
| 意图分类 | LLM 分类 | 比规则匹配更准确，支持自然语言意图理解 |
| 工具设计 | 聚合工具 | 减少工具数量，提高 Agent 选择正确工具的概率 |
| 审批流程 | 交互式确认 | 更自然的用户体验，避免中断后等待轮询 |
| 会话存储 | DB + Redis | 持久化历史消息，Redis 缓存热点会话和检查点 |

## Success Criteria

### 功能指标

1. **AI 响应准确率 ≥ 90%**
   - 意图理解正确
   - 工具调用正确
   - 结果呈现清晰

2. **交互式确认覆盖率 100%**
   - 所有高风险操作触发确认
   - 用户可选择确认/取消/查看详情

3. **工具调用成功率 ≥ 95%**
   - 参数解析正确
   - 执行无错误
   - 结果返回完整

### 性能指标

| 指标 | 当前 | 目标 |
|------|------|------|
| 首次响应时间 | 3-5s | < 2s |
| 简单问答响应 | 3-5s | < 1s |
| 会话历史加载 | 2-3s | < 500ms |
| 前端包大小 | 未优化 | < 200KB (gzip) |

### 代码质量指标

| 指标 | 当前 | 目标 |
|------|------|------|
| 前端最大文件 | 71KB | < 20KB |
| 后端路由数 | 47 | 6 |
| 工具数量 | 30+ | 10 |
| 测试覆盖率 | ~60% | > 80% |

## Risks & Mitigations

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 前端重写工作量大 | 开发周期长 | 分阶段实施，先核心功能再扩展 |
| 混合模式意图分类错误 | 简单请求走了 Agent 流程 | 添加 fallback，分类错误时自动降级 |
| 工具合并影响兼容性 | 现有调用方需要适配 | 保留旧工具 API，标记为 deprecated |
| 交互式确认流程复杂 | 状态管理困难 | 使用 eino 原生的中断恢复机制 |

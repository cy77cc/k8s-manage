# Proposal: 服务管理职责边界重构

## Summary

重新定义服务管理的职责边界，明确服务配置与部署目标的关系，优化服务详情页的交互体验。

## Motivation

当前服务管理模块存在以下问题：

1. **职责不清**: 服务详情页的"配置"Tab 包含"部署目标"卡片，但部署目标不应属于服务配置的范畴
2. **交互混乱**: 服务列表页点击"编辑配置"只显示提示，没有实际功能；点击服务名称和"查看详情"都跳转到详情页
3. **部署流程不合理**: 预设"默认部署目标"的设计不灵活，服务可能有多种部署场景
4. **集群环境约束缺失**: 集群没有环境属性，无法约束服务只能部署到匹配环境的集群

## Goals

1. 明确服务管理的职责边界：**只关注服务配置，不关注部署目标**
2. 删除服务详情页"配置"Tab 中的"部署目标"卡片
3. 将"编辑配置"Modal 的内容直接作为"配置"Tab 展示（内联编辑）
4. 给集群添加环境约束字段，用于限制服务部署的目标集群
5. 修复服务列表页的交互问题

## Non-Goals

1. CI/CD 功能设计（后续单独讨论）
2. 日志和监控功能开发（待后续规划）
3. 部署流程的具体交互设计（本次只涉及服务管理边界）

│  不关注:                                                        │
│  ├─ 默认部署目标 (删除 Service.default_target_id 等字段)         │
│  └─ 部署时的集群/命名空间选择 (部署时临时选择)                    │
│                                                                 │

│                         部署流程设计                             │

│  └──────────────────┘    └─────────────────┘                   │

│                      集群环境约束设计                            │

│                                                                 │
│  Cluster 新增字段:                                               │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │  env_type: development | staging | production             │ │

│  部署时会检查两者是否匹配                                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

### UI 变更

#### 服务详情页 - 配置 Tab 重构

**Before:**
```
配置 Tab
├─ 部署目标卡片 (cluster_id, namespace, deploy_target)  ← 删除
├─ 环境变量集卡片
└─ 渲染输出预览
```

**After:**
```
配置 Tab
├─ 服务配置卡片 (原"编辑配置"Modal 的内容，改为内联编辑)
│   ├─ 基本信息: 名称、环境、负责人、状态
│   ├─ 运行时配置: runtime_type, config_mode, render_target, service_kind, service_type
│   ├─ 标签
│   └─ 配置内容 (标准配置 JSON / 自定义 YAML)
├─ 环境变量集卡片
└─ 渲染输出预览
```

#### 服务列表页 - 交互修复

**Before:**
- 点击服务名称 → 跳转详情页
- 点击"查看详情" → 跳转详情页
- 点击"编辑配置" → 只显示 message.info 提示

**After:**
- 点击服务名称 → 跳转详情页
- 点击"查看详情" → 跳转详情页
- 点击"编辑配置" → 跳转详情页并定位到配置 Tab

## Implementation Plan

### Phase 1: 数据模型变更

1. **Cluster 表添加 env_type 字段**
   - 新增迁移文件添加 `env_type VARCHAR(32) NOT NULL DEFAULT 'development'`
   - 更新 Cluster model

2. **Service 表清理 (可选，后续再做)**
   - 标记 `default_target_id`, `default_deployment_target_id` 为废弃
   - 后续版本迁移后删除这些字段

### Phase 2: UI 重构

1. **服务详情页重构**
   - 删除"配置"Tab 中的"部署目标"卡片
   - 将"编辑配置"Modal 改为"配置"Tab 的内联编辑模式
   - 保留"环境变量集"卡片和"渲染输出预览"

2. **服务列表页修复**
   - "编辑配置"改为跳转到详情页的配置 Tab

### Phase 3: 部署 API 调整

1. **部署接口变更**
   - `POST /services/:id/deploy` 必须传入 `cluster_id` 和 `namespace`
   - 后端校验 Service.env 与 Cluster.env_type 是否匹配

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| 现有数据迁移 | 现有集群默认设置 env_type='development'，手动调整生产集群 |
| 现有部署流程中断 | 部署目标从服务配置中分离，但部署 API 保持兼容 |
| 用户习惯改变 | 通过 UI 引导说明新的部署流程 |

## Success Criteria

1. 服务详情页"配置"Tab 不再包含"部署目标"卡片
2. 集群表有 `env_type` 字段
3. 服务列表页"编辑配置"按钮功能正常
4. 部署时可以约束服务只能部署到匹配环境的集群

## Context

### 当前状态

```
┌─────────────────────────────────────────────────────────────────┐
│                      现有 Token 刷新流程                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   请求 ──▶ API ──▶ 401/4005/4006 ──▶ 尝试刷新 ──▶ 重试请求      │
│                                                                 │
│   问题：用户会感知到请求失败（loading、error flash）             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**现有实现**:
- `web/src/api/api.ts`: 有 `tryRefreshAndRetry` 方法，在请求失败时刷新
- `web/src/components/Auth/AuthContext.tsx`: 管理 token 状态，但未监听刷新事件
- 后端 `/auth/refresh`: 已实现，支持 token 轮换

**约束**:
- 后端 JWT 配置: accessToken 过期时间较短（如 15min），refreshToken 较长（如 7d）
- refreshToken 存储在 Redis 白名单中，支持撤销

## Goals / Non-Goals

**Goals:**
1. 实现 token 主动刷新：在 token 即将过期前主动刷新，用户无感知
2. 状态同步：刷新成功后 AuthContext 的 token 状态同步更新
3. 并发安全：多个请求同时触发刷新时，只发起一次刷新请求
4. 失败处理：刷新失败时清除状态并跳转登录页

**Non-Goals:**
- 不修改后端 token 机制（后端已完善）
- 不实现 remember me 功能（属于登录策略）
- 不处理多标签页同步（后续可扩展）

## Decisions

### D1: 刷新时机 - 定时检查 vs 过期预判

**选择**: 过期预判 + 定时检查组合策略

```
┌─────────────────────────────────────────────────────────────────┐
│                      Token 生命周期                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   登录 ───────────────────────────────────────────────▶ 过期    │
│            │                              │                     │
│            ▼                              ▼                     │
│        [过期前5min]                   [过期时]                  │
│        主动刷新                       被动刷新（兜底）           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**策略**:
- 解析 JWT 获取过期时间（不依赖额外 API）
- 在过期前 5 分钟主动刷新
- 定时器每分钟检查一次（降低 CPU 占用）

**备选方案**:
- 方案 A: 固定间隔刷新（如每 10 分钟）→ 不智能，可能频繁刷新
- 方案 B: 每次请求检查 → 过于频繁，性能影响
- 方案 C: 过期预判 + 定时检查（选择）→ 平衡性能和及时性

### D2: 刷新触发位置 - ApiService vs AuthContext

**选择**: 独立 TokenManager + AuthContext 监听

```
┌─────────────────────────────────────────────────────────────────┐
│                      架构设计                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌──────────────┐     事件      ┌──────────────┐              │
│   │ TokenManager │ ───────────▶  │ AuthContext  │              │
│   │              │   tokenRefresh│              │              │
│   │ - 定时检查   │               │ - 更新状态   │              │
│   │ - 主动刷新   │               │ - 同步存储   │              │
│   └──────────────┘               └──────────────┘              │
│          │                                                      │
│          │ 调用                                                  │
│          ▼                                                      │
│   ┌──────────────┐                                              │
│   │  ApiService  │                                              │
│   │              │                                              │
│   │ - refresh()  │                                              │
│   │ - 拦截器     │                                              │
│   └──────────────┘                                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**理由**:
- TokenManager 职责单一：管理 token 生命周期
- AuthContext 专注状态管理
- 通过事件解耦，便于测试和扩展

### D3: 并发刷新控制

**选择**: 使用现有 ApiService 中的 `refreshPromise` 机制

```typescript
// 已有实现，无需修改
private refreshPromise: Promise<boolean> | null = null;

async refreshAccessToken(): Promise<boolean> {
  if (this.refreshPromise) {
    return this.refreshPromise; // 复用进行中的刷新
  }
  // ...
}
```

### D4: 刷新失败处理

**选择**: 清除状态 + 跳转登录页 + 事件通知

```
刷新失败 ──▶ 清除 localStorage ──▶ 发送 tokenExpired 事件 ──▶ 跳转登录页
```

**理由**:
- 统一的登出流程
- 用户可以看到明确的登录提示
- 保留当前路径，登录后可恢复

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| JWT 解析失败（格式错误） | 捕获异常，降级到被动刷新 |
| 刷新请求失败 | 清除状态，跳转登录页 |
| 定时器内存泄漏 | 在 AuthProvider unmount 时清理 |
| 多标签页同时刷新 | refreshPromise 机制保证只刷新一次 |
| 系统时间不准确 | 使用服务器时间（可从 JWT 的 iat/expire 计算） |

## Migration Plan

### 实施步骤

1. 创建 `TokenManager` 工具类
2. 增强 `ApiService`，暴露刷新成功/失败事件
3. 更新 `AuthContext`，监听 token 刷新事件
4. 添加单元测试

### 回滚策略

- 移除 TokenManager，恢复原有被动刷新逻辑
- 移除 AuthContext 中的事件监听
- 改动小，回滚风险低

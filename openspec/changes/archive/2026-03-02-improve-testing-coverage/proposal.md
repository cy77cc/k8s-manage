# Proposal: Improve Testing Coverage

## Summary

建立完整的测试体系，提升后端测试覆盖率从 ~10% 到 60%+，前端测试覆盖率从 ~10% 到 50%+，并引入集成测试和 E2E 测试框架。

## Motivation

当前测试现状：
- 后端：219 个 Go 文件，仅 21 个测试文件（覆盖率 ~9.6%）
- 前端：114 个 TS/TSX 文件，仅 11 个测试文件（覆盖率 ~9.6%）
- 缺少集成测试、E2E 测试、性能测试、安全测试

测试不足导致：
- 重构风险高，难以保证行为一致性
- Bug 容易遗漏到生产环境
- 新功能缺乏回归保护

## Scope

### In Scope

1. **后端单元测试扩展**
   - CICD 模块测试
   - RBAC 模块测试
   - Monitoring 模块测试
   - Host 模块测试

2. **后端集成测试框架**
   - 测试工具包 (testutil)
   - Mock SSH/K8s/AI 客户端
   - HTTP Handler 测试

3. **前端单元测试扩展**
   - API 客户端测试（认证、Token 刷新、错误处理）
   - 核心页面测试补齐
   - 工具函数测试

4. **E2E 测试基础设施**
   - Playwright 配置
   - 关键用户旅程测试

5. **测试基础设施**
   - 覆盖率报告
   - CI/CD 集成
   - Makefile 测试命令

### Out of Scope

- 性能/负载测试（后续单独处理）
- 安全渗透测试（后续单独处理）
- 混沌测试（后续单独处理）

## Success Criteria

- [ ] 后端测试覆盖率 >= 60%
- [ ] 前端测试覆盖率 >= 50%
- [ ] 核心业务流程有集成测试覆盖
- [ ] 关键用户旅程有 E2E 测试覆盖
- [ ] CI/CD 中有测试覆盖率门禁
- [ ] 所有新测试通过

## Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| 测试编写耗时 | 中 | 分阶段实施，优先核心模块 |
| Mock 实现复杂 | 中 | 抽象接口，渐进式 Mock |
| E2E 测试不稳定 | 低 | 添加重试机制，选择稳定场景 |

## Timeline

- Phase 1: 测试基础设施（1 周）
- Phase 2: 后端核心模块测试（2 周）
- Phase 3: 前端核心测试 + E2E 框架（2 周）

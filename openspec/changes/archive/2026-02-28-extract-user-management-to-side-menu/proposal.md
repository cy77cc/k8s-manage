## Why

当前“用户管理”相关能力被放在“系统设置”下，信息架构与操作路径不匹配，导致高频账号治理操作被埋深，且权限边界不清晰。随着平台多角色协作增加，需要将用户、角色、权限管理能力独立成右侧菜单入口，并按角色进行可见性控制，降低误操作并提升管理效率。

## What Changes

- 将“管理用户”“管理用户角色”“管理权限”从“系统设置”中拆分，迁移为独立的右侧导航菜单分组。
- 新增菜单可见性策略：仅具备指定角色（或等效权限）的用户可见并可进入该菜单。
- 统一用户、角色、权限三个页面的信息架构与交互规则（列表、筛选、详情/编辑、授权反馈）。
- 对无权限用户提供一致的降级体验：隐藏入口，且直接访问时返回明确的无权限提示。
- 明确前后端权限契约：前端负责菜单与页面级可见性，后端负责接口级强校验与审计留痕。

## Capabilities

### New Capabilities
- `user-access-governance`: 统一用户、角色、权限管理的独立导航入口与页面流程，覆盖账号治理主操作闭环。
- `role-aware-navigation-visibility`: 基于角色/权限声明控制右侧菜单展示与访问路径，确保“可见即有权、无权即不可见且不可调用”。

### Modified Capabilities
- `platform-capability-baseline`: 增补平台管理类导航与权限控制基线要求，使“系统设置”与“访问治理”职责分离。

## Impact

- Frontend
  - 受影响模块：`web/src/layout`（右侧菜单渲染）、`web/src/router`（路由守卫）、`web/src/pages`（用户/角色/权限页面）。
  - 新增/调整 API 调用：`web/src/api/modules` 下用户、角色、权限相关接口聚合与权限错误处理。
- Backend
  - 受影响模块：`internal/service` 中用户、角色、权限与 RBAC 校验逻辑；Casbin 策略查询与授权校验链路。
  - API 影响：`/api/v1` 下用户管理相关接口需要补充角色可见性与访问控制约束。
- Security & Audit
  - 无权限访问需统一 403 语义与审计字段，确保操作可追踪。
- Delivery
  - 需要新增 OpenSpec delta（新能力 + 基线能力修改）并通过 `openspec validate --json`。
